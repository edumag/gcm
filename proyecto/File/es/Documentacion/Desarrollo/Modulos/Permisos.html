La gestión de los permisos se realiza según roles, el rol administrativo que es el que tiene el administrador por defecto, permite cualquier acción.Eventos es el encargado de pedir permisos para todas las acciones que no esten definidas en los archivos de eventos_usuario.php de los módulos.

Algunas acciones que no estan contempladas en eventos_usuario.php pero que deben tener acceso abierto, pueden definirse en el archivo eventos_usuario.php incluyendolas a la lista blanca de eventos.

En el módulo 'menu' podemos ver un ejemplo:

modulos/menu/eventos_usuario.php

// añadimos metodo menu_ajax a la lista blanca, para que no se necesiten 
// permisos para lanzarlo.

$this->set_lista_blanca('menu','menu_ajax');

Los eventos definidos en 'eventos_admin.php' en los módulos tendran que pasar la verificación antes de ser lanzados.

Configuración por defecto

Roles por defecto:

  • administrador: Se le permiten todas las acciones.
  • usuario: Limitado
  • editor: Puede añadir contenido propio
  • traductor: Se le permite añadir o modificar traducciones

Formas de uso

permiso('editar_contenido', 'modulo',TRUE);

En este caso la acción a realizar esta relacionada con la edición de contenido del módulo 'contenido', en caso de que el usuario actual no tenga permiso se enruta con aviso de que no tiene permisos.

El valor TRUE del segundo parametro es el que nos indica que debe enrutarse en caso de tener permisos, si no pasamos este parametro simplemente la función nos devolvería un FALSE y seriamos nosotros los que gestionasemos la respuesta.

 Configuración

Dentro de los archivos 'eventos_admin.php' de los módulos podemos definir a que roles pertenecen las acciones de los módulos.

Ejemplo en módulo literales

/**
 * Configuración para los permisos de usuario
 *
 * permitimos administrar literales a editores y traductores
 */

$acciones['literales']['panel_literales'][] = 'editor';
$acciones['literales']['panel_literales'][] = 'traductor';
